# Eval-стенд почерка (vision-пайплайн распознавания шагов)

Дев-инструмент владельца — прогоняет размеченные фото решений через
`classify_step_photo` и считает метрики качества. НЕ прод-эндпоинт.

## 1. Как собрать набор реальных фото

- **30-50 фото** тетрадей разных детей: разный почерк, разные ручки, разное
  освещение (окно/лампа/вспышка), клетчатая и линованная бумага.
- **Один шаг в кадре** — то же самое, что видит модель в проде на step-submit
  (не всё решение целиком, а один конкретный шаг лесенки).
- Кириллица + цифры — как в реальной тетради (не печатный текст).
- Кладите файлы в `backend/data/eval/photos/` (JPEG/PNG, как есть с телефона).

## 2. Как заполнить `labels.csv`

Формат — 4 колонки:

```csv
photo_file,decomp_idx,step_n,expected_verdict
photo_001.jpg,0,1,match
photo_002.jpg,3,2,mismatch
photo_003.jpg,10,1,match
```

- `photo_file` — имя файла в `photos/` (без пути).
- `decomp_idx`, `step_n` — какой шаг снят. Смотрите в БД:
  `SELECT decomp_idx, n, instruction_ru, expected_value FROM problem_steps WHERE decomp_idx = <idx>;`
- `expected_verdict` — **ваша** разметка (что на самом деле на фото): `match`
  (ребёнок написал верно) или `mismatch` (ребёнок ошибся на этом шаге).
  Стенд сам возьмёт `instruction_ru`/`expected_value`/условие задачи из БД —
  вам нужно только сфотографировать и указать правду.

⚠️ `instruction_ru`/`expected_value`/`statement` (условие задачи) стенд
резолвит из БД сам, вам их вписывать не нужно.

## 3. Как запустить стенд

```bash
cd backend
../.venv/bin/python scripts/eval_stand.py
```

Полезные флаги:

```bash
--dir backend/data/eval     # где искать photos/ + labels.csv (default)
--dsn <postgres-dsn>        # какая БД (default: dev nismathbot)
--provider gemini|openai    # переопределить провайдера (default: из .env)
--limit 10                  # прогнать только первые N примеров (быстрый прогон / жёсткий rate-limit)
--delay 5                   # пауза между вызовами в секундах (Gemini free-tier душит 429 — увеличивайте при ошибках)
--out results.csv           # куда писать per-example лог (default: results.csv рядом с labels.csv)
```

## 4. Как читать метрики

В конце прогона стенд печатает:

- **`accuracy`** — доля верных предсказаний среди тех, где модель дала чёткий
  вердикт (`match`/`mismatch`); `unsure` в знаменатель НЕ входит — это не про
  точность, это отдельный исход («модель не уверена, попроси перефотать»).
- **`false_reject_rate`** — **ГЛАВНОЕ число для нас.** Доля случаев, когда
  ребёнок написал шаг ВЕРНО, а модель сказала «неверно» (`mismatch`). Это
  худший UX-сценарий: ребёнка обвиняют в ошибке, которой не было. Держим его
  ниже `false_accept_rate` — пропустить реальную ошибку менее вредно, чем
  ложно наказать за верный шаг (принцип из VISION.md).
- **`false_accept_rate`** — обратная величина: ребёнок ошибся, а модель
  сказала «верно» (пропуск ошибки). Хуже, чем ничего, но менее болезненно
  для доверия ребёнка к системе, чем false-reject.
- **`unsure_rate`** — доля «не уверена, перефотай». Не ошибка классификатора,
  а UX-трение: если она большая — модель часто просит пересъёмку, это
  раздражает, но не вводит ребёнка в заблуждение.
- **confusion-матрица 2×3** (ожидание × предсказание, + средний confidence по
  ячейке) — детализация, откуда взялись agregированные rate выше.

Per-example лог (`results.csv`) — по одной строке на фото: что ожидали, что
предсказала модель, confidence, что модель "увидела" (`seen_value`), и была
ли строка пропущена (`skipped`/`error`) из-за 429/сетевой ошибки.

## 5. Как сравнить провайдеров

```bash
../.venv/bin/python scripts/eval_stand.py --provider gemini  --out results_gemini.csv
../.venv/bin/python scripts/eval_stand.py --provider openai   --out results_openai.csv
```

Сравнивайте `false_reject_rate` и `accuracy` между прогонами — это решает,
какой провайдер держать основным (`VISION_PROVIDER` в `.env`).

## Файлы (не коммитятся — см. `.gitignore`)

- `photos/` — реальные фото детей (персональные данные).
- `labels.csv` — разметка (не коммитим синтетическую версию; свою разметку
  тоже не коммитить — там имена файлов из вашей коллекции).
- `results*.csv` — прогоны (воспроизводимы запуском стенда).

Синтетический smoke-набор — `scripts/gen_eval_synthetic.py` (8 сгенерированных
фото по реальным ступеням из БД, только для проверки что пайплайн жив, НЕ для
замера точности на реальном почерке).
