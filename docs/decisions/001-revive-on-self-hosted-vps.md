# 001: Воскрешение kodi-web на self-hosted VPS, не на Railway

**Дата:** 2026-06-22
**Статус:** accepted

## Контекст
Проект делался в learning-эпоху, был задеплоен на Railway (single-Docker + Railway Postgres), последний коммит 2026-03-02. Railway-проект локально не залинкован, CLI разлогинен — состояние старой инфры неизвестно. Нужно воскресить и снова поднять рабочим для личной дальнейшей разработки.

## Решение
Деплоить на **свой VPS** (тот же сервер, что sibling-проект CDP: ssh-хост `aiplus`), через **Docker Compose**: сервис `app` (kodi-web образ) + собственный `postgres`-контейнер, host nginx vhost наружу, отдельные порты (8200/5434 заняты CDP). Стартуем со **свежей БД** (старые Railway-данные тестовые, не переносим — это обходит migration-gap по FSRS-колонкам, AUDIT MIG-1).

## Альтернативы
- **Остаться на Railway** — отклонено: владелец явно хочет self-hosted (контроль, стоимость), CDP уже мигрировал Railway→VPS, есть отработанный паттерн.
- **Перенести старую Railway-БД** — отклонено: данные тестовые, перенос тянет migration-gap и лишний риск; свежая БД проще и чище.

## Последствия
- Нужен Docker Compose + docker/nginx по образцу CDP (нет в репо — создаётся в фазе 3).
- nginx vhost/SSL требует root (Умид) — внешняя зависимость; до vhost доступ через SSH-туннель.
- Деплой — rsync + `docker compose build/up` на сервере (не railway up, не git push).
- Стек kodi-web (SQLAlchemy async + asyncpg, Flutter) сохраняется — переписывать под CDP-стек (psycopg3/React) НЕ будем.
