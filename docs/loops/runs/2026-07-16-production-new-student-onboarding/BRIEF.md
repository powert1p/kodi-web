# Production onboarding нового ученика

## Задача

Новый ученик открывает публичный продукт, регистрируется по телефону и PIN и без промежуточного
dashboard начинает первый содержательный урок. Повторный вход так же сразу возобновляет текущий
урок. Старый Flutter flow не должен быть публичной точкой входа.

## Product truth

- Аудитория: ребёнок 10–13 лет, который впервые открыл приложение после занятия в центре.
- Главный job: начать учиться, а не разбираться в функциях приложения.
- Первый урок: серверный маршрут `mixtures-1`, пример → опоры → самостоятельность → перенос.
- Прогресс, ответы и возобновление остаются серверными; browser storage хранит только JWT.
- Фото, review, AI-чат и диагностика остаются вторичными recovery-инструментами и не входят в
  первый путь нового ученика.
- Детская регистрация не запрашивает родительское согласие на фото. Оно появляется отдельным
  parent handoff только перед первой фото-зависимой функцией и не блокирует обычный урок.

## Frozen decision

Выбран direct-to-lesson flow:

1. `/`, `/app` и `/app/` приводят к одному React PWA на `/app/`.
2. Гость попадает на регистрацию/вход.
3. После успешной регистрации или входа клиент получает текущий серверный учебный путь и сразу
   открывает его `primary_action.lesson_id`.
4. Если путь временно недоступен или пуст, аккаунт не теряется: ученик попадает на `/app/`, где
   видит восстановимое loading/error/empty state.
5. Повторный login возобновляет серверный текущий урок; `/app/` остаётся безопасным fallback,
   если определить урок временно невозможно.

Отклонены:

- обязательная диагностика до первого урока — противоречит прямому старту;
- отдельный welcome/dashboard после регистрации — добавляет пустой шаг;
- жёстко зашитый lesson id в auth UI — создаёт второй источник истины;
- сохранение старого Flutter UI на `/app` или `/` — сохраняет два разных продукта.

## Register lock

Регистр: **собранный**.

Запрещены dashboard-card soup, конкурирующие primary actions и onboarding ради объяснения UI.
Математика и текущий шаг должны появиться раньше любых вспомогательных возможностей.

## State matrix

- guest/new phone;
- phone-check loading/error;
- registration name/grade/PIN;
- registration loading/error/duplicate race;
- successful registration + available lesson;
- successful registration + path request error/empty;
- first lesson loading/active/error;
- reload/relogin resume;
- canonical entry `/`, `/app`, `/app/`;
- mobile 375×844 and desktop 1280×900;
- keyboard, reduced motion, offline/network failure.

## Scope

Maintenance внутри принятого v11 визуального канона. Меняется информационная архитектура входа и
переход регистрации в обучение; новый визуальный redesign не требуется.
