# Спека: слой тем Common Core над графом навыков (view-only)

> Дата: 2026-06-24 · /brainstorming · продолжение CC-restructure (2026-06-23). Опирается на docs/specs/cc_topic_skill_tree.json/.md.

## Цель
Сейчас «подтемы» графа — плоские 15 тегов, чисто UI-фильтр; CC-иерархия (43 темы, 61 ребро) живёт только в docs/specs и с прод-графом кодом не связана. Сделать «подтему» настоящим уровнем графа: **Раздел→Тема→навык**, чтобы ученик видел структуру и связи между темами вместо плоского списка тегов.

## Scope
- Новые сущности: `topics` (43: код CC/НИШ, strand, ru/kz-название, класс, порядок) + `topic_edges` (61 ребро-пререквизит между темами).
- `nodes.topic_id` — привязка каждого из 118 узлов к ровно одной теме (мост 118→43).
- API `/api/graph/me`: к каждому узлу добавить `topic_id`; новые массивы `topics` (ru/kz-названия, рёбра, классы, strand) и `strands` (порядок). Существующие поля не трогаем.
- Фронт `graph_page.dart`: вместо `byTag` — иерархия Раздел→Тема→навык (аккордеон), прогресс на каждом уровне, тема показывает свои темы-пререквизиты (информативно).
- Идемпотентный bootstrap в `run.py`: новые таблицы (create_all) + `ADD COLUMN topic_id` + upsert тем/рёбер + `UPDATE topic_id` — работает и на свежей, и на существующей задеплоенной БД.

## Out-of-scope
- НЕ трогаем 118 узлов, 2525 задач, mastery, attempts, BKT — всё сохраняется как есть.
- НЕ меняем движок: диагностика/практика/экзамен/fringe/zones работают как раньше — **темы только вид, не гейтят путь** (выбор владельца).
- НЕ вносим 337 микро-навыков как узлы графа (другой грейн — отдельная история).
- `static/graph.html` (Telegram-экспорт) — не трогаем.
- DROP/DELETE/TRUNCATE — запрещено.

## Продуктовые решения (из ответов)
- **Слой тем НАД навыками** (118 остаются). **Иерархия — только вид графа**, движок не трогаем.

## Технические решения (мои, обоснование в строку)
- Грейн темы = CC-кластер (43 темы). Strand = домен CC (9: RP/EE/NS/G/MD/OA/NF/NBT/SP) + НИШ (1) = 10 разделов; вывожу из кода темы — баланс детализации/читаемости.
- Маппинг 118→43 генерирую агентом по названиям узлов + CC-дереву, верифицирую выборку; каждый узел → ровно одна тема, low-confidence помечаю — готового моста нет, имена богатые.
- Темы рендерю только непустые (узлов ≥1); CC-коды в UI скрываю, показываю ru/kz-название — вайб-кодеру «6.RP.A» в UI не нужен (код хранится, но не на виду).
- Схема additive, Alembic нет → новые таблицы `create_all` + idempotent `ALTER`/upsert в `run.py` (существующий паттерн) — работает на задеплоенной БД без migration-gap.
- `topic_id` без FK-констрейнта в `ALTER` (на существующей БД); целостность держат seed + тест-инварианты.
- Данные: новый `backend/data/cc_topics_v01.json` (topics + edges + node→topic map), сгенерён из cc_topic_skill_tree.json детерминированно + агент-маппинг.

## Критерии успеха (проверяемые)
- Инвариант данных (pytest): 118/118 узлов имеют валидный `topic_id` (0 сирот); 43 темы; 61 ребро; каждая тема в strand; рёбра ссылаются на существующие темы.
- API `/graph/me` возвращает `topics[]`, `strands[]` и `topic_id` на каждом узле; старые поля целы (контракт-тест).
- Идемпотентность: повторный прогон `seed_topics` не падает и не плодит дублей (тест на двойной прогон).
- `flutter analyze` 0 errors; `flutter build web --release` проходит.
- Playwright: страница графа показывает Раздел→Тема→навык, прогресс на уровнях, без ошибок в консоли (скриншот).
- Выборочная сверка 12–15 маппингов узел→тема — осмысленны.
- `.arb` ru/kz: новые строки в обоих файлах, 0 хардкода.

## Риски
- Кривой маппинг узел→тема → агент-верификатор + ручная сверка выборки; low-confidence помечены, правка тривиальна (данные в JSON).
- Существующая задеплоенная БД → тест на двойной прогон seed; ALTER идемпотентны.
- Фронт-модель и контракт API разъехались → контракт-тест + flutter analyze + Playwright.
