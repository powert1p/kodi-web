# Спека: кабинет «Работа над ошибками» — MVP (фронт, mock-данные)

> Дата: 2026-06-23 · /brainstorming · Опирается на ресёрч (стек+UX+визуал) и существующую модель тема→микро-навык→задача-в-шагах (fractions_steps_v1.json / full_decomposition_v1.json).

## Цель
Ребёнок написал интенсив-срез (например 40 вопросов), часть с ошибками. Кабинет = слой работы над ошибками: каждую ошибочную задачу разбираем по **подзадачам (шагам)**, ведущим к пониманию БЕЗ выдачи ответа; на провале — подсказка-лесенка, затем задача **попроще** (тот же микро-навык, ступень ниже); рядом — **AI-тьютор** для диалога. Сейчас — MVP фронта на mock-срезах, потом докрутка + интеграция с бэком.

## Scope (MVP)
- Новое веб-приложение (отдельно от Flutter), mobile-first 414px, акцент на мобилку.
- 3-4 экрана ядра флоу: **Срез-хаб → Разбор задачи (лесенка подзадач) → AI-тьютор (bottom sheet) → Закрытие/празднование**.
- Mock-данные среза «Дроби · интенсив» (форма как fractions_steps_v1.json: steps[{n, instruction_ru, micro_skill, expected_value}] + catalog).
- KaTeX-рендер математики. Тирный ввод ответа (для MVP — текст + live-превью KaTeX; кастомный keypad — докрутка).
- Дизайн-токены (шрифты/цвет/радиусы/моушн) как единый источник, по дизайн-направлению ниже.

## Out-of-scope (якорь)
- НЕ трогаем backend, схему БД, Flutter-фронт (старое не удаляем — снос/cutover отдельным заходом по явному согласию).
- НЕ строим реальные эндпоинты (декомпозиция/тьютор/закрытие) — только контракт по факту, данные мокаем.
- НЕ WebView-интеграция сейчас (стек выбран embeddable; мост — докрутка).
- Голос/фото-ввод (Mathpix), родительский экран, учительское назначение — не v1.

## Продуктовые решения (решил сам по ресёрчу; владелец может отменить)
- Закрытие ошибки = разобрал подзадачи **И** сдал 1 свежую проверочную (та же нода, новые числа, без подсказок).
- Тьютор Socratic: не выдаёт ответ, но всегда есть выход «зайдём проще» (никаких тупиков).
- Тон: ошибка — не приговор, а «где растёт мозг»; хвалим усилие, не способности; без красного shake.

## Технические решения (мои, обоснование в строку)
- **Стек:** React 19 + TS strict + Vite (SPA, без SSR) + Tailwind v4 + Zustand (сессия) + KaTeX — лёгкий статичный билд, быстрый TTI на слабом Android, тот же билд → PWA и позже WebView (`flutter_inappwebview`, токен через JS-мост, не URL).
- **Данные mock** в `src/mock/` по форме реального разбора — чтобы потом подменить на API без переделки UI.
- **Hash-routing** — чистый deep-link внутри WebView без серверных round-trip.
- Контент задач/шагов/чата — Markdown + `$...$`, рендер через KaTeX.

## Дизайн-направление (анти-AI-slop, под design-скиллы)
- Шрифты (self-host woff2): **Lexend** (тело/UI, читаемость у детей), **Bricolage Grotesque** (заголовки/празднование), **Space Grotesk** (табличные цифры: 32/40, проценты, счётчики). НИКОГДА Roboto/Inter/Arial.
- Цвет (CSS-переменные): якорь синий `#2563EB`, blue→indigo градиент шапки, ОДИН тёплый акцент (coral) только для «починил/празднуем», светофор mastery: amber=почти / teal=освоено / slate=к разбору. Фон НЕ плоский: слоистый тёплый градиент + лёгкое свечение за активной подзадачей, 3 ступени теней.
- Signature: **вертикальная «Лесенка понимания»** — ступени-подзадачи зажигаются по очереди; при провале ступени вставляется ступень попроще НИЖЕ, затем подъём обратно.
- Моушн: staggered-reveals на входе, spring на верном, мягкий warm-wobble на ошибке (никогда красный shake), конфетти на закрытии. Маскот-математик (округлый, большие глаза) как личность тьютора/празднований.

## Критерии успеха (проверяемые)
- `npm run build` проходит, `npm run dev` поднимается; tsc strict без ошибок.
- 3-4 экрана работают на mock: срез→задача→лесенка с gating→подсказка/проще→тьютор-шит→закрытие.
- Рендерится в Playwright на мобильном вьюпорте (390-414px) без console-ошибок; скриншоты подтверждают дизайн-направление (шрифты не Roboto, фон не плоский, лесенка-signature видна).
- Прошёл ui-design-review / nielsen-эвристики ≥7/10.

## Риски
- Tailwind v4 + self-host шрифтов + KaTeX в Vite — конфиги; замечу на `npm run build` + скриншоте (FOUT = шрифт не подключился).
- Over-design vs MVP — держать ядро флоу, не плодить экраны; докрутка потом.
