# Промпты движка авторинга в стиле Карпатого (essence + definition of done)

**Дата:** 2026-06-24 · **Статус:** in progress

## Цель
Промпты автора/критика лесенок сейчас «тяжёлые»: жёсткие worked-примеры (GOOD/BAD), пошаговый
алгоритм «СНАЧАЛА ПОДУМАЙ 1/2/3». Это даёт шаблонность: задача h долбит одну мысль («на N» vs «в N
раз») 3 раза подряд, задача g набита филлер-арифметикой до единственной реальной развилки. Переписать
промпты по Карпатому: дать СУТЬ + **definition of done** (проверяемые критерии), убрать примеры и
рецепт — модель сама находит форму, а DoD держит планку.

## Принцип (Karpathy, goal-driven)
«Define success criteria. Loop until verified.» Не диктовать КАК — задать ЦЕЛЬ + проверяемый DoD и
крутить, пока критерии не выполнены. В нашей архитектуре это уже есть: **автор** = суть + DoD (свобода
формы), **критик** = тот же DoD как верификатор, цикл автор→критик→правка = «loop until verified».
Мэппинг на принципы из CLAUDE.md: simplicity-first → «целит в развилку, не в счёт» (чинит g);
no-abstraction-single-use → «каждая идея один раз» (чинит h); goal-driven → DoD + цикл.

## Scope
- Переписать `cabinet/engine/author-prompt.md`: суть + DoD (критерии как ТЕСТЫ) + свобода формы +
  выходной JSON. Выкинуть банк примеров и нумерованный алгоритм.
- Переписать `cabinet/engine/critic-prompt.md`: тот же DoD как набор проверяемых свойств. Добавить два
  свойства, которых не хватало: **filler/simplicity** (ступень = рутинный счёт → флаг, чинит g) и
  **cross-rung single-idea** (одна идея в нескольких ступенях через разную обёртку → дубль, чинит h).
- Синхронизировать инлайн-копии AUTHOR/CRITIC в `cabinet/engine/ladder-engine.workflow.js`.
- Перегенерировать g+h движком на новых промптах; i — как regression/diversity-якорь (был лучшим).

## Out-of-scope
- Не трогаю выходную JSON-схему (формат, не рецепт) и grounding-блок. `reasoning` в схему НЕ
  возвращаем (урок про truncation-loop). effort=medium.
- Не переавторю a/c/e/f и не гоняю весь банк 2525 — отдельный заход по решению пользователя.
- Не меняю рантайм/кабинет-компоненты — только данные g/h в `srez.ts` после валидации.

## Продуктовые решения (вопросов не было — бриф ясен)
- Ноль worked-примеров, ноль пошагового алгоритма: критерии DoD сформулированы как операц. ТЕСТЫ
  («убери ступень — путь к пониманию рушится?»), это даёт калибровку без примера-образца.
- Авторинг лесенок — это prompt-craft, его пользователь явно доверил мне; пишу промпты сам, НЕ через
  subagent-driven (это для код-фич), чтобы не потерять калибровку. Гейты те же.

## Критерии успеха (проверяемые)
- Новый **g**: нет филлер-ступеней (рутинный `80÷(10/3)` уходит); модель `собств ± течение` строится
  ДО вопроса про разрыв; лесенка доводит до 4 км/ч.
- Новый **h**: «на N» vs «в N раз» свёрнут в ≤2 ступени (не 3 подряд); лесенка доходит до x=21.
- **i** не регрессирует (остаётся чистым: fencepost + формула, доводит до 98550).
- Три лесенки РЯДОМ — разной формы (нет общего скелета); критик ловит подсунутый филлер/дубль на тесте.
- `npx tsc --noEmit` чисто; Playwright mobile: g/h/i рендерятся, choose/compute работают, 0 ошибок в консоли.
- Движок без loop: `grep -c StructuredOutput` по агентам = 1–3.

## Риски
- Без примеров модель может «поплыть» по гранулярности «одна мысль» или чаще писать leading →
  больше правок критика (рост латентности). Замечу по числу revised + по находкам критика. Митигация:
  DoD-критерии заданы как тесты + критик ловит и крутит (это и есть «loop until verified»).
- Критик без примеров может стать мягче. Замечу: подсуну в валидации заведомо-филлерную/дублирующую
  ступень мысленно — новый критерий должен её назвать.
