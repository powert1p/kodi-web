# Deep Research: AI-native реализация — как сделать, как хранить, как сейчас подключают ИИ

> Дата: 2026-06-22 · Режим: deep (7 параллельных агентов + red-team) · ~70 источников · триангуляция ≥2 источника/вывод
> Вопросы владельца: (1) как реализовать step-декомпозицию, (2) как это хранить, (3) как сейчас ИИ подключают к таким процессам — «новая эра, а не как раньше программисты».

---

## Главный вывод одной строкой

**«Новая эра» НЕ означает «LLM делает всё». Каждая выжившая production-система (Khanmigo, MathGPT, Duolingo Max) сошлась на ГИБРИДЕ: LLM делает язык (разбор задачи на шаги, подсказки, тьюторинг, объяснения), а детерминированный код (SymPy/калькулятор + заранее размеченный контент) решает правильность.** Ровно то, что ты хочешь автоматизировать сильнее всего — «спроси LLM вживую, на каком шаге ребёнок ошибся» — это **самое слабое и опасное звено** (42-65% точности даже с готовым решением; харм бьёт по слабым ученикам — твоей аудитории). Решение: разметить шаги заранее (LLM офлайн + SymPy-проверка + твоё ревью), а в рантайме LLM лишь **сопоставляет** ответ с готовыми шагами, не ищет ошибку с нуля.

---

## Executive Summary

Три независимых ресёрч-агента сошлись к одной архитектуре. **Где LLM реально незаменим и production-проверен — это качественный контентный слой:** разобрать задачу на шаги и пометить каждый атомарным микро-навыком (эксперты предпочитают LLM-теги человеческим в ~2/3 случаев [KC-2024]); сгенерировать подсказки и worked-examples (в RCT не отличимы от подсказок живого репетитора [hint-RCT]); смоделировать «виртуальных учеников» для оценки сложности задач без большой когорты. **Где LLM ненадёжен и не должен решать — математическая правильность:** локализация первого неверного шага даёт лишь 42-55% сырьём и 49-65% даже с эталонным решением [EMNLP-2509.01395]; открытая диагностика заблуждений — macro-F1 <0.5 у всех 18 моделей, ~58% ложных приписываний [diag-2504.00843].

Производственный ответ — **архитектура Khanmigo**, перенесённая на solo-масштаб: LLM для языка/педагогики, детерминированный калькулятор перед каждым утверждением о правильности, история ученика + непройденные пререквизиты подаются простым текстом, всё опирается (grounding) на заранее заготовленный контент. Khan получил +3.4% от истории и +2.7% от пререквизитов в контексте; grounding в человеческие шаги был «крупнейшим рычагом точности».

**Стоимость — не проблема:** на Haiku 4.5 ($1/$5 за M токенов) с prompt-caching (90% скидка на кэш) разбор+подсказка на одну ошибку — менее $0.01; сессия из 20 задач (6 ошибок) ≈ $0.035; разовая разметка всех 2525 задач через Batch API (50% скидка) ≈ десятки–$190 один раз; месячный расход на одну когорту — единицы долларов. **Реальные ограничения — задержка и галлюцинации диагноза, а не деньги.** Quizlet Q-Chat закрыли именно из-за live-генерации на каждого юзера — твой кэш + предразметка этого избегают.

**Mastery держим простым счётчиком, НЕ LLM:** специализированный KT/счётчик бьёт LLM-как-предсказатель-mastery при том же или лучшем качестве в 615-12,400 раз дешевле и в 12 раз быстрее [KT-vs-LLM-2026]. LLM работает над КОНТЕНТОМ (шаги, теги, подсказки), счётчик — над СОСТОЯНИЕМ.

## 1. Как реализовать — архитектура (гибрид)

### Сквозной цикл (LLM в петле ТОЛЬКО при ошибке)

```
1. Ребёнок решает задачу P (шаги размечены заранее, офлайн).
2. Детерминированная проверка финального ответа (SymPy) →
   ВЕРНО: ++счётчики микро-навыков задачи, Leitner планирует повтор, готово. БЕЗ LLM.
3. НЕВЕРНО → LLM-вызов #1 (structured output): сопоставляет работу ученика
   с ГОТОВЫМИ шагами P, возвращает {failed_step, micro_skill, misconception, confidence}.
   SymPy перепроверяет шаг — при расхождении правильность решает SymPy, не LLM.
4. --счётчик нужного микро-навыка, записываем misconception.
5. LLM-вызов #2 (или из кэша): worked sub-step для упавшего шага + 1 задача
   полегче на тот же навык (выбор SQL-фильтром по micro_skill).
6. Leitner ставит навык на интервальный повтор.
```

### Где LLM, а где детерминированный код

| LLM (язык/педагогика) | Код (правильность/состояние) |
|---|---|
| Офлайн-разбор задач на шаги + теги микро-навыков (разово) | Правильность ответа и шагов — **SymPy** (бесплатно, мгновенно, 100% надёжно) |
| При ошибке: сопоставить работу с готовыми шагами + misconception-нарратив | Инкремент счётчиков, ratio mastery |
| При ошибке: текст подсказки / worked-example | Планировщик повторов (Leitner/FSRS) |
| Socratic-тьюторинг (не выдавать ответ) | Выбор следующей задачи (SQL по micro_skill) |
| | **Весь happy-path (верные ответы) — без LLM** |

**Правило:** LLM никогда не решает правильность и никогда не считает расписание.

### Ключевой приём надёжности: классификация, а не локализация

Открытый вопрос «на каком шаге ошибка?» = ~30% F1 (почти подбрасывание монетки на многошаговых). Поэтому в рантайме LLM **выбирает** из заранее размеченных и SymPy-проверенных шагов (closed-set classification), а не ищет ошибку с нуля. Ещё лучше: если UI собирает ответы по шагам — **первый шаг, чья SymPy-проверка падает, И ЕСТЬ упавший шаг, вообще без LLM**; LLM лишь объясняет уже найденную ошибку.

## 2. Как хранить — схема Postgres (5-7 таблиц, без BKT, без графа)

Это стандартная модель ITS из CMU DataShop (Q-matrix), совпадает с прошлым решением:

```
micro_skills        (id, code, name_ru, grade_band, parent_id)   -- закрытый словарь ~30-60 атомарных навыков; parent_id = сжатый граф
problems            (id, statement, answer_canonical, grade, difficulty_prior)
problem_steps       (id, problem_id, step_index, instruction_ru, expected_expression, micro_skill_id)  -- предразметка
step_skills         (step_id, skill_id)                          -- Q-matrix M:N (если шаг тренирует >1 навык)
attempts            (id, student_id, problem_id, submitted, is_correct, created_at)  -- НЕИЗМЕНЯЕМЫЙ event-log
step_diagnoses      (id, attempt_id, failed_step_id, micro_skill_id, misconception, llm_confidence, symbolic_agreed)  -- аудит
student_skill_state (student_id, micro_skill_id, n_seen, n_correct, mastery_ema, last_seen_at, leitner_box, due_at)  -- mastery-счётчик + расписание
remediations        (id, problem_step_id, misconception, worked_text, created_at)  -- кэш подсказок: одинаковые ошибки переиспользуют текст
student_profile     (student_id, summary_ru ~150 слов, weak_skills jsonb, refreshed_at)  -- LLM-читаемое саммари из счётчиков
```

**Два слоя — намеренно:** `attempts` (неизменяемая правда, append-only) + `student_skill_state` (производный кэш). Если завтра лучшая модель переразметит шаги или расширится словарь навыков — переигрываешь `attempts` и пересобираешь состояние. Mastery — EMA: `mastery = 0.3·is_correct + 0.7·old` (recency и забывание бесплатно).

**AI-native слой хранения:** НЕ заменять таблицы вектор-базой. Современные memory-системы (Mem0/Zep) держат структурную таблицу как источник правды + короткое LLM-читаемое саммари в промпт. Для тебя: генеришь ~150-слов `summary_ru` из счётчиков, кладёшь в кэшированный промпт. `pgvector`/эмбеддинги — отложить, пока когорта не вырастет.

## 3. Как сейчас подключают ИИ — продукты и паттерны «новой эры»

**Сошедшийся паттерн (все выжившие продукты):** LLM-язык + детерминированная правильность + grounding в свой контент.

| Продукт | Что делает | Урок для kodi |
|---|---|---|
| **Khanmigo** (Khan Academy) | GPT-тьютор; **отдельный калькулятор** для арифметики (не LLM); подтягивает человеческие шаги/решения ПЕРЕД ответом | Главный шаблон. Grounding + калькулятор = крупнейшие рычаги точности |
| **MathGPT.ai / Wolfram** | LLM объясняет, **символьный движок вычисляет и проверяет** | SymPy как оракул правильности |
| **Duolingo Max** | На неверный ответ шлёт GPT-4 структурный контекст, LLM **только объясняет**, не судит | Самое безопасное и дешёвое — копировать |
| **MATHia / Squirrel AI** | Хардкод-граф 10-30k узлов + Bayesian, 25 лет данных | **Избегать** — нужны миллионы учеников для калибровки (ты ниже data-floor) |
| **Quizlet Q-Chat** | LLM-тьютор — **закрыт**: live-генерация на юзера не окупилась | Кэш + предразметка спасают от этой ловушки |
| **IntelliCode** (open MIT) | 6 LangGraph-агентов над versioned learner-state, FastAPI | Взять ОДНУ идею (single-writer state как счётчик), **не** рой агентов |

**Что реально «новая эра» и стоит ставить СЕЙЧАС** (production-проверено, дёшево, в твоей зоне продукта):
1. LLM офлайн-разбор + теги микро-навыков (эксперты предпочитают LLM-теги в 2/3 случаев) — **заменяет хардкод-граф структурой, которую эксперты выбирают чаще человеческой**.
2. LLM-подсказки/worked-examples по запросу, с кэшем по (задача, упавший шаг) — в RCT не хуже живого репетитора.
3. LLM-симуляция «виртуальных учеников» для оценки сложности задач — **бутстрап сложности без большой когорты** (твоя ситуация).

**Что НЕ брать (хайп для solo на малом масштабе):**
- Live «спроси LLM, какой шаг неверный» без каркаса (42-65%).
- Открытые misconception-ярлыки ребёнку (macro-F1 <0.5).
- Multi-agent рои (LangGraph/ArangoDB), обучение PRM, фитинг BKT, вектор-БД как основа, runtime self-consistency.
- LLM-как-предсказатель-mastery (в 615-12400× дороже при нулевом приросте).

## 4. Стоимость и задержка (не ограничение)

| | Стоимость |
|---|---|
| Верный ответ (happy path) | **$0** — только SymPy |
| Ошибка: диагноз + подсказка (Haiku + кэш) | **< $0.01** (ближе к $0.001) |
| Сессия 20 задач (6 ошибок) | **~$0.035** |
| Разовая разметка всех 2525 задач (Batch −50%) | **$10–190 один раз** (зависит от модели/sampling) |
| Месяц на одну когорту | **единицы долларов** |

Рычаги: **Haiku 4.5** по умолчанию ($1/$5), **prompt-caching** (кэшируем статичный скелет задачи → 90% скидка на вход), **Batch API** (разовая разметка −50%). Эскалация на Sonnet только если на eval-наборе Haiku мис-тегает. Задержка: happy-path — мгновенно (SymPy), ошибка — 1-3с на вызов (приемлемо, бывает только после неверного ответа). **Единственная денежная ловушка — runtime self-consistency** (умножение вызовов); не нужна — эталонный ответ уже хранится.

## 5. Guardrails — чтобы ИИ не научил ребёнка неверной математике (слоями)

Это критично: харм бьёт по слабым ученикам (твоя аудитория), которые **меньше всех способны заметить неверную поправку** [EDM-2025-RCT, 252 ученика].

1. **SymPy авторитетен по правильности** — детерминированный калькулятор перебивает LLM. Самый дешёвый и решающий слой.
2. **Grounding** — в каждый промпт кладём заготовленные шаги + эталонный ответ; LLM рассуждает над известно-верным, а не выводит с нуля.
3. **Классификация, не локализация** — LLM выбирает из готовых шагов, не ищет ошибку открыто.
4. **Закрытый набор misconception** — ярлыки заблуждений только из малого, тобой проверенного списка, привязанного к шагам; **никогда не free-form ребёнку**.
5. **Confidence-гейтинг** — при низкой уверенности ИЛИ расхождении с SymPy → нейтральная подсказка «давай перепроверим этот шаг вместе», без обвинения в конкретной ошибке.
6. **Hint-лестница** (5 уровней) + системный промпт «не выдавать ответ» + diff-проверка, что live-смягчение тона не добавило новых чисел.
7. **Held-out eval-набор** (50-100 known-wrong попыток, размеченных тобой) — мерить точность диагноза ПЕРЕД доверием когорте и при каждой смене модели/промпта (паттерн Khanmigo-дашборда). **Неотключаемый ремень безопасности.**

> Вердикт red-team: слои ВМЕСТЕ делают «неверная математика дошла до ребёнка» почти-невозможным событием, а не вероятностным — но **убери любой слой (особенно SymPy или закрытый misconception-набор) — и пол проваливается.**

## 6. План сборки (MVP-first, для solo-нетехнаря)

**Не варить океан сразу на 2525 задач. Сначала вертикальный срез на одном семействе навыков (напр. дроби, ~30-50 задач).**

- **Шаг 0 — eval-набор (~1 день):** руками 20-30 задач с известными неверными ответами. Это линейка точности для всех будущих изменений.
- **Шаг 1 — словарь микро-навыков:** ОДИН LLM-проход по выборке задач кластеризует кандидатные атомарные навыки → ты куришь закрытый список ~30-60. **Это гейтящий артефакт человеческого суждения — твоя зона.** ⚠️ Нельзя выдумать сверху: статичная таксономия = 27-36% точности тегов, эмпирически-выведенная = 74-83%.
- **Шаг 2 — офлайн-пайплайн разметки (Batch API)** по срезу: разбор на шаги (strict JSON), тег из закрытого списка, hint-лестница, SymPy-проверка КАЖДОГО числового шага (reject при расхождении), твой spot-check.
- **Шаг 3 — схема Postgres:** event-log `attempts` + тонкий счётчик. Без BKT.
- **Шаг 4 — MVP БЕЗ runtime-LLM:** SymPy-проверка ответа + счётчики + Leitner + SQL-выбор задач. **Это уже работающее адаптивное приложение с НУЛЕВЫМ runtime-LLM-риском.**
- **Шаг 5 — live-диагноз при ошибке:** LLM-вызов #1 (structured output, confidence-гейт, кросс-чек SymPy). Шипишь только когда eval-точность приемлема.
- **Шаг 6 — live-ремедиация:** LLM-вызов #2 (worked-step + задача полегче) с кэшем `remediations`.
- **Шаг 7 — Socratic-тьюторинг** (промпт Khanmigo/LearnLM) — multi-turn, фаза 2.
- **Шаг 8 — только после рабочего среза:** прогон офлайн-разметки на остальные 2525.

## 7. Честные оговорки (red-team)

1. **Словарь микро-навыков — реальное узкое место, не код.** Выводить из LLM-прохода по задачам, потом курировать. Если выдумать сверху — всё ниже по течению мис-роутится.
2. **Не доверять top-1 LLM-тегам как истине** (35-56% точного совпадения) — spot-check словаря и 5-10% выборки обязателен.
3. **SymPy может не покрыть все типы** (текстовые задачи с нечисловыми шагами рассуждения). Прежде чем считать «100% проверяемо» — прогони выборку сложных NIS-задач. (Medium-уверенность.)
4. **Scope creep назад к live-разбору** — постоянный соблазн «пусть LLM сделает вживую, чтобы не размечать». Все три агента флагают это как путь, который рано или поздно научит ребёнка неверному методу. Гейт «разметить заранее» — неотключаемый.
5. **Дрейф качества при смене модели** — без eval-набора надёжность тихо гниёт при апгрейде Claude/смене промпта.

**Уверенность:** ВЫСОКАЯ на ядро (3 независимых агента сошлись на одной гибридной архитектуре + одном центральном предостережении: live-локализация ошибки = слабое звено). СРЕДНЯЯ на точную стоимость разметки и полноту покрытия SymPy — ни то, ни другое не меняет порядок сборки.
