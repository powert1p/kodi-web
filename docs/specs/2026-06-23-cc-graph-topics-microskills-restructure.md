# Спека: граф Common Core = темы, 372 микро-навыка под ними (дедуп + слияние)

> Дата: 2026-06-23 · /brainstorming · Ultracode (Workflow). Опирается на docs/specs/2026-06-22-ready-math-graph-foundation.md + full_micro_skill_catalog.md.

## Цель
Сейчас 372 микро-навыка лежат плоским списком по 15 «доменам», граф (kodi 118 нод) и навыки живут отдельно. Владелец решил: **граф Common Core = канонический стандарт тем**, микро-навыки вешаем под него. Нужна чистая 3-уровневая иерархия с готовыми стрелками-пререквизитами + честная чистка навыков (дубли убрать, «про одно и то же» слить), без потери покрытия задач.

## Scope
- 3-уровневая структура: **STRAND** (крупная тема, ≈15 доменов в чистых рус-названиях) → **TOPIC** (CC-кластер, напр. `4.NF.B`, это узел графа со стрелками) → **МИКРО-НАВЫК** (наши атомы, лист).
- Каждый выживший навык → ровно ОДИН CC-стандарт (напр. `4.NF.B.3`); из кода выводятся cluster(topic) и domain(strand).
- Дедуп + слияние 372 (внутри домена и кросс-домен): убить точные дубли, слить синонимы, перенести мисфайлы (ряды в geometry → logic).
- NIS-ветка для того, чего у CC нет на 2-7 классах (логика/комбинаторика/теория чисел/олимп-методы): свои topic-узлы `NIS.*`, привязка к ближайшему CC-стандарту ребром «builds-on».
- Подключить готовые **541 CC-ребро** как граф пререквизитов между topic/standard.
- Отчёт о слияниях: что→во что, чтобы ни один навык не пропал молча.

## Out-of-scope
- Маппинг КТП-уроков на эту структуру — отдельный заход (это teaching-order overlay поверх CC-backbone).
- Не трогаем прод-код, схему БД, миграции, kodi 118 нод (новая структура — рядом, старое не удаляем).
- Не строим тренажёр/рантайм. Не меняем full_decomposition_v1.json (разметка задач остаётся; навыки только переименовываются через карту слияний).
- DROP/DELETE/TRUNCATE данных — запрещено.

## Продуктовые решения (бриф ясен, вопросов не было)
- Граф = общий стандарт по темам (CC), навыки — под темами. Дубли убрать, «про одно и то же» — слить. Качество важнее числа.

## Технические решения (мои, обоснование в строку)
- **Грейн topic = CC-кластер** (`4.NF.B`), tag навыка = CC-стандарт (`4.NF.B.3`) — кластер читаемая «тема» (~40-60 реально задетых), стандарт даёт точность маппинга.
- **Слияние консервативное:** объединять только подлинные синонимы/дубли/мисфайлы, педагогически различное не схлопывать (каталог уже 902→372, агрессивный мердж потеряет сигнал).
- **Карта слияний `merged_into`** обязательна — full_decomposition остаётся валиден через rename, ноль потерянных задач (инвариант: Σ freq сохраняется).
- **NIS-ветка отдельным strand** — CC 2-7 не покрывает олимпиадную логику/комбинаторику/тер.числа, насильно мапить нельзя.
- **Оркестрация:** Workflow — pipeline по 15 доменам (агент дедупит+мапит свой срез на CC), затем consolidation-агент (кросс-домен дубли, сборка дерева, проверка инварианта). Сборка файлов — детерминированно в Python после.

## Критерии успеха (проверяемые)
- `cc_topic_skill_tree.json`: strand→topic(CC code+label+prereq edges)→[навыки]; каждый навык имеет cc_standard | NIS-код; 0 навыков без родителя.
- **Инвариант покрытия:** Σ freq навыков после == Σ freq до (3684+ шагов); каждый из 372 исходных кодов либо выжил, либо есть в `merged_into` (0 «исчезнувших»).
- Дубли-мисфайлы (`arithmetic_sequence_term`/`sum_arithmetic_sequence` в geometry, `count_in_arithmetic_sequence` в word_problems) — слиты в logic-аналоги.
- Граф связный: каждый topic имеет ≥0 рёбер из CC; NIS-topic привязан builds-on к CC-якорю.
- Читаемый `cc_topic_skill_tree.md`: итоговое число навыков, число тем, число слияний, дерево по strand.

## Риски
- Агент мапит навык не в тот CC-стандарт → consolidation-агент + мой ревью спорных; репрезентативная сверка 10-15 маппингов руками.
- Кросс-домен дубль не пойман (один агент видит свой срез) → отдельная стадия глобального дедупа по label-эмбеддингу/ключам.
- NIS-навыков окажется много (логика 41 + часть divisibility) → ок, это и есть НИШ-специфика; помечаю явно, не прячу.
