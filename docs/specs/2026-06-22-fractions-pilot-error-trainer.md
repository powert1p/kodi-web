# Спека: пилот «Работа над ошибками + тренажёр» на теме ДРОБИ

> Дата: 2026-06-22 · Брейншторминг · Опирается на docs/specs/2026-06-22-research-ai-native-implementation.md

## Цель
Тесты (срез/воскресный/интенсив) уже есть и хранят только балл. Сверху строим **слой работы над ошибками + тренажёр**: взять заваленные вопросы → разобрать по шагам → найти упавший **микро-навык** → дать дополнительные задачи именно на него. Пилотируем на одной теме — **дроби** — чтобы обкатать модель и показать «что с графом».

## Модель данных: 3 слоя (ответ на «что с графом / как темы хранятся»)
1. **Тема (topic)** = единица КТП = FR01…FR13. Что преподаёшь и что меряет срез. Крупная. 118 уже есть.
2. **Микро-навык** = атомарное умение (сократить дробь, общий знаменатель, умножить дроби, порядок действий…). Мелкий, **переиспользуемый между темами**, ~15-18 на дроби. **ЭТО единица mastery** (счётчик на него).
3. **Задача → шаги** = каждая задача разбита на шаги, шаг помечен ОДНИМ микро-навыком.

**«Граф» сжимается до:** тема → её микро-навыки (это тегирование, не DAG) + тонкая цепочка пререквизитов между микро-навыками (1-2 уровня, авторская). **Арифметика = слой foundational-микро-навыков**, всплывающих внутри задач многих тем — ровно «арифметика уходит как под-тема».

## Scope (пилот)
- Каталог микро-навыков дробей (~15-18), выведенный из реальных 248 дробных задач.
- Декомпозиция дробных задач на шаги-с-микро-навыками + SymPy-проверка числовых шагов.
- Дизайн слоя «разбор ошибки → упавший микро-навык → доп. задачи (тренажёр)».
- Демонстрация полного цикла на 2-3 реальных задачах.

## Out-of-scope (якорь от расползания)
- НЕ пересобираем тесты (срез/воскресный/интенсив остаются как есть, хранят балл).
- НЕ соединяем всё во флоу сразу (явно сказано владельцем).
- Runtime-вход «на каком вопросе ошибся» даёт существующее приложение / айтишники — наш слой поверх.
- КТП (не нужен сейчас), остальные 117 тем, BKT, граф-движок, multi-agent — вне scope.

## Продуктовые решения (из ответов владельца)
- Тесты = только оценка; разбор+тренажёр = отдельный слой поверх.
- Старт — дроби; арифметика трактуется как микро-навык, не тема.
- КТП пока не подключаем.

## Технические решения (мои, обоснование в строку)
- 3-слойная модель тема/микро-навык/задача-шаги — разрешает «арифметика=под-навык», убивает 118-нодовый DAG.
- mastery = счётчик student×микро-навык (n_seen/n_correct), без BKT — ниже data-floor (см. ресёрч).
- Декомпозиция офлайн (LLM размечает, SymPy проверяет каждый числовой шаг, владелец ревьюит), заморозка в Postgres; рантайм только сопоставляет ответ с готовыми шагами — иначе LLM ловит шаг лишь в 42-65% (ресёрч).
- Тренажёр: упавший микро-навык → задачи, где он primary, на ступень легче (bounded-ability).
- Каталог микро-навыков ВЫВОДИТСЯ из реальных задач, не выдумывается сверху (статичная таксономия = 27-36% точности тегов vs выведенная 74-83%).

## Критерии успеха (проверяемые)
- Каталог 15-18 микро-навыков дробей; каждый встречается ≥2 реальных задачах.
- ≥10 реальных дробных задач (разных FR-тем) декомпозированы: каждый шаг → ровно 1 микро-навык; числовой результат шага проходит SymPy-проверку (== эталон).
- На 2-3 примерах показан полный цикл: неверный ответ → упавший шаг → микро-навык → конкретная доп. задача того же навыка.
- Многошаговые темы (FR09 комбинированные, FR12 этажные) собираются ИЗ микро-навыков, своего отдельного навыка не заводят — демонстрация «сложное = композиция простого».

## Риски
- SymPy не покроет нечисловые/текстовые шаги (word problem FR11) → такие шаги помечаем `check=llm_only`, пайплайн не падает; замечу при прогоне выборки.
- LLM-декомпозиция режет криво → ревью обязателен; начинаем с ручной сверки 13 тем (по 1 задаче), потом батч.
- Микро-навык дробит слишком мелко/крупно → ревизия каталога после первых 13 задач; держать ~15-18, не плодить.
