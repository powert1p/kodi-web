## Как работать без Fable (правила эпохи)

1. **Каждый заход начинай с чтения** VISION.md (правки продукта) или DESIGN_SYSTEM.md (правки UI) — они канон; при конфликте «идея в чате» vs «канон в доке» — сначала обнови канон осознанным решением, потом делай.
2. **Дизайн-задачи Opus'у ставить ТОЛЬКО по рельсам** (см. раздел «Дизайн-протокол» ниже): копирование эталонных компонентов, токены-only, скриншот-гейт. Новую ЭСТЕТИКУ Opus не изобретает — она зафиксирована.
3. **Гейты неизменны**: pytest (реальный PG) + vitest + build + Playwright-скриншот до «done»; деструктив данных — только с явного согласия владельца; wrap в конце содержательного куска.
4. **Если задача «мутная» и не бьётся на ясные куски** — это сигнал отложить её до консультации (одна Fable-сессия по надобности) или разбить на меньшие эксперименты, а не давать Opus'у карт-бланш.

## Сценарные развилки (куда пойдёт — решения по данным, не по настроению)

**A. Итог пилота (после 1.3):**
- Дети пользуются, работы честные → масштабируем Этап 1 на всех учеников центра, идём в Этап 2 (данные/файнтюн).
- Дети бросают ("ГДЗ быстрее") → пивот в **центр-режим**: поэтапная сдача на планшетах В ЦЕНТРЕ под присмотром (Этап 3 раньше Этапа 2), дом — только повторение.
- Массовое списывание → форсировать anti-copy: вариативные параметры задач (генерация вариантов из банка) + контрольные в центре; «домашние» вердикты помечать как неверифицированные.

**B. Юрист по данным (параллельно 1.0):**
- Облако ОК с согласием → продолжаем на Gemini.
- Трансграничная передача запрещена → self-host vision-модель раньше плана (Qwen-VL на VPS/GPU-аренда) — тогда Блок 1.1 сразу меряет и self-host вариант.

**C. Точность на eval-наборе (после 1.1):**
- ≥85% → идём по плану.
- 60-85% → сузить MVP: только числовые шаги (без рисунков/геометрии), UI «перепиши шаг разборчиво в рамку» (структурированный лист).
- <60% → фолбэк-режим: система показывает эталонный следующий шаг, ребёнок самопроверяется (Photomath-паттерн) — теряем автопроверку, сохраняем поэтапность; копим данные дальше.

**D. Лимиты Opus/подписки:** дизайн-задачи упираются → откладывать в бэклог «до Fable-сессии», не жечь заходы на переделки; механика (бэк/тесты/данные) — Opus справляется по рельсам.

## Дизайн-протокол для Opus (research: docs/specs/2026-07-03-research-ai-design.md)

**Суть**: slop лечится не промптом, а сужением выбора. Эстетика зафиксирована в `webapp/DESIGN_SYSTEM.md` (v5) — Opus её НЕ пересматривает, только исполняет. «Инструкция не переживает свежий контекст — переживает то, что не проходит гейт».

**Как ставить UI-задачу (шаблон промпта имплементеру):**
```
Прочти webapp/DESIGN_SYSTEM.md (канон). Задача: <экран/компонент>.
Эталон паттерна: <файл-эталон из §3>. Используй токены/компоненты: <список>.
Целевое состояние: <скриншот или ASCII-макет — композицию задаёшь ТЫ в задаче, не исполнитель>.
DoD (бинарный): [ ] lint:design/vitest/build зелёные [ ] один primary-CTA виден без скролла
[ ] запреты §2 не нарушены [ ] скриншот 390×844 приложен [ ] состояния loading/error/empty есть.
```

**Приёмка — всегда отдельным критик-вызовом** (самопроверка в том же контексте теряет 20-40 п.п.): второй агент получает скриншот + §2/§4 канона и отвечает по бинарному чеклисту. 1-2 итерации по КОНКРЕТНОЙ критике («CTA ниже фолда — подними», не «сделай красивее»), после 2-3 — плато, стоп.

**Чего Opus'у НЕ поручать в дизайне**: придумать экран/эстетику с нуля; менять токены/шрифты; вводить новые компоненты без строки в §3 канона. Такие хотелки — в бэклог «до Fable-сессии» (сценарий D).

**Гейты в коде**: `npm run lint:design` (не-токен hex/px/полушаги/шрифты → exit 1) + Playwright-скриншоты против `webapp/design-baselines/` + axe smoke. Дрейф-аудит — раз в месяц.
