# VISION — от фото листочка к live-тьютору

> Обновлено: 2026-07-03. Это постоянный контекст проекта: любое продуктовое решение сверяется с этим документом.
> Обоснование каждого тезиса — deep research `docs/specs/2026-07-03-research-photo-to-live-tutor.md`.

## Продукт одной строкой

Оффлайн математический центр, где **тему объясняет живой учитель, а всю индивидуальную практику ведёт AI**, который видит ход решения каждого ребёнка и помогает в момент ошибки — сначала по фото листочка, в перспективе — вживую на планшете, вплоть до голосового помощника.

## Бизнес-рамка

- **Учитель** — объяснение нового и точечное вмешательство. **AI** — рутина: проверка работ, диагностика пробела, подбор следующей задачи. Продаём «AI освобождает учителя от проверки тетрадей», НЕ «AI заменяет учителя» (так выжили все референсы: Squirrel AI, пост-регуляторный Китай).
- **MVP**: дети практикуются дома, фотают листочки с решением ПОЭТАПНО — на этом собираем первые данные и первый работающий цикл ценности.
- **Дифференциатор**: мы проверяем РАБОТУ ученика и находим, ГДЕ он ошибается — противоположность переполненному рынку «сфоткал → получил ответ» (ГДЗ-сервисы; в Китае сектор снесён регулятором как вредный). Педагогическая ставка: пошаговый тьюторинг d=0.76 против d=0.31 у проверки ответа — но полный эффект требует ЖИВОЙ петли (эффект async-фото скромнее; растёт по этапам к live). Честная оговорка: MVP не детектит списывание — «anti-copy» строим постепенно (вариативные параметры задач, контрольные в центре), до этого не обещаем.
- **Данные — второй продукт**: датасетов детского рукописного мат-решения не существует; каждый проверенный шаг = размеченный пример. Через сотни–тысячи примеров — собственная модель (нужна ради приватности и независимости, не экономии: API стоит копейки).

## Технические принципы (несущие, не менять без нового research)

1. **Классификация, не транскрипция.** Никогда не «прочитай, что написал ребёнок», всегда «совпадает ли написанное с ожидаемым шагом из банка». Grounding по verified-шагам — главный рычаг точности (87% ошибок моделей — чтение, не суждение).
2. **Правильность решает код** (check_answer/sympy), LLM — язык и педагогика. Fingerprints реальных ошибок — первичный детектор, LLM-судейство — fallback (LLM локализуют чужую ошибку лишь на ~55-65%).
3. **Мягкий вердикт.** «Ложное неверно» на правильном решении для ребёнка хуже пропуска ошибки → порог к мягкости, при неуверенности — «покажи ещё раз» / эскалация, не бинарное «неверно».
4. **Одна архитектура на все этапы.** Фото → сегментация → классификация шагов → сократик. Канвас-снапшоты планшета и голос — тот же пайплайн, другие входы/выходы. Инвестиции MVP не выбрасываются.
5. **Согласие родителя ДО первого сбора** (закон РК №94-V, ст.145 ГК; ориентир COPPA/GDPR-K), отдельный чекбокс «для обучения модели». ⚠️ Шире consent: №94-V требует локализации данных в РК — отправка детских фото в облачный vision-API (сервера вне РК) = трансграничная передача → консультация юриста ДО запуска сбора; это же двигает self-host модель раньше по роадмапу.

## Роадмап

### Этап 0 — Ядро тренажёра ✅ (в проде, 2026-07)
Фото готового решения → Gemini-диагноз (grounded) → чат-тьютор → закрытие ошибки контрольной. Граф знаний v02 (114 узлов, чистые пререквизиты), банк 2525 задач с verified-шагами, fingerprints, «мои проблемные темы».

### Этап 1 — Поэтапная сдача (= MVP стартапа) ⟵ СЛЕДУЮЩИЙ
Ребёнок решает по шагам и фотает работу по ходу (кусок листочка после шага/блока — не каждый шаг отдельным фото: churn). Сервер сегментирует, классифицирует против expected_value шага, ловит уход в сторону ДО финала.
- **Первый шаг — behavioral-пилот на детях центра** (red-team: бизнес-риск выше технического): реальные семьи, метрики «вернулся на вторую сессию» и ручной просмотр работ на списывание — валидируем спрос и честность ДО/параллельно технике.
- Параллельно: **собственный eval-набор** (реальные тетради: кириллица + клетка + детский почерк — зоны без единого бенчмарка) — мерить до обещаний точности; на нём же финальный выбор провайдера (Gemini = дефолт, Mathpix/Qwen-VL — контроль).
- Consent-флоу родителя + конвейер сбора (фото+шаг+вердикт) + юрист по локализации/трансграничной передаче.
- Выдаём задачи, покрытые decomp-банком; не-матчащийся шаг → LLM-fallback с пометкой низкой уверенности; «не уверен» → «перефотай» или «разбор в центре» (не «спроси учителя» — его дома нет).
- Anti-gaming: лог паттернов подсказок с первого дня (hint-abuse крадёт ⅓ обучения). Anti-copy: вариативные параметры задач — как только банк это позволит.
- Гейт этапа: ≥85% верных вердиктов на eval-наборе И false-reject ≤5%; ребёнок проходит задачу поэтапно end-to-end; ≥N семей центра вернулись на вторую неделю.

### Этап 2 — Данные и качество
Разметка накопленного, метрики качества вердиктов, файнтюн-эксперимент (сотни–тысячи примеров + синтетика — порог подтверждён research), динамика для родителя («было 12 проблемных шагов → осталось 3»).
- Гейт: файнтюн бьёт базовый Gemini на нашем eval-наборе ИЛИ осознанно остаёмся на API.

### Этап 3 — Планшет (центр и/или дом)
Рукописный ввод на канвасе (PWA тянет стилус: pressure, 240Hz, palm rejection — native не требуется до упора), периодические снапшоты канваса → тот же пайплайн. Здесь рынок ПУСТ: никто не делает push-вмешательство по ходу письма.
- Поведенческие сигналы (пауза/зачёркивание) — триггер частоты проверки, не детектор.
- Гейт: вмешательство «ушёл не туда» срабатывает до завершения решения с приемлемой точностью на eval-наборе чернил.

### Этап 4 — Live-тьютор
Проактивное вмешательство в реальном времени + голосовой слой (дом, наушники; ASR детей 10-13 — переходная зона; <13 лет = zero-data-retention или self-host). Учительский дашборд «кто где застрял» — поздний этап (решение владельца).

## Топ-риски и как замечаем (порядок = приоритет)

1. **Спрос/retention/чит дома без учителя рядом** — удержит ли продукт ребёнка, когда ГДЗ даёт ответ за 3 секунды → behavioral-пилот на семьях центра ПЕРВЫМ шагом этапа 1; ручной аудит работ на списывание.
2. **Кириллица+клетка без бенчмарка** → свой eval-набор на этапе 1, до него точность не обещаем.
3. **Ложные «неверно» на верном решении** (в т.ч. валидный путь вне банка) → метрика false-reject отдельно, порог к мягкости, LLM-fallback + расширение банка альтернативными путями из потока.
4. **Юр-локализация** (данные детей РК → зарубежный API) → юрист до запуска сбора; план self-host раньше при негативном ответе.
